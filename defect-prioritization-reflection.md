🏆 Milestone: Reporting & Tracking Issues Effectively
⚡ Tracking and Prioritising Defects
🎯 Goal
Understand how to categorize, track, and prioritize defects based on severity and impact.

❓ Why is this important?
Not all bugs are equal—some are critical showstoppers, while others are minor inconveniences. Knowing how to prioritise defects correctly helps ensure that high-impact issues get fixed first while lower-priority ones don’t block development.

🔍 Research & Learn

**What is the difference between bug severity and bug priority?**
Bug severity and bug priority are different concepts. Severity describes how serious the technical impact of the bug is on the system. For example, a crash, data loss, or login failure would be high severity. A small UI misalignment would usually be low severity. Priority describes how quickly the bug should be fixed from a business perspective. A bug may be technically minor but high priority if it affects an important client or a key feature before a release.

**How do teams decide which bugs to fix first? (Business impact, risk, effort, etc.)**
Teams decide which bugs to fix first based on several factors: impact on users, business risk, number of users affected, security concerns, release deadlines, and development effort. Critical issues that block users, cause data loss, or affect revenue are usually fixed first. Sometimes a lower-severity issue can be prioritized higher if it affects a core feature or an upcoming demo.

**What tools can QA use to track defects in an Agile workflow?**
In Agile workflows, QA typically tracks defects using tools like Jira, Azure DevOps, ClickUp, or similar issue-tracking systems. These tools allow teams to assign severity, priority, status, and link bugs to specific sprints or releases. They also help maintain transparency and communication across the team.

**What happens when QA, developers, and product teams disagree on priority?**
When QA, developers, and product teams disagree on priority, the discussion should focus on impact and risk. QA explains user impact, developers explain technical complexity, and product managers decide based on business goals. The final priority is usually a shared decision, but product owners often have the final say in Agile teams.

📝 Reflection

**How would you prioritise a cosmetic UI issue vs. a login failure?**
I would prioritize a login failure much higher than a cosmetic UI issue. A login failure blocks users from accessing the app, so it is high severity and high priority. A cosmetic issue does not block functionality, so it would normally be lower priority unless it affects branding or user trust in an important flow.

**If a developer says a bug is too minor to fix, how would you respond?**
If a developer says a bug is too minor to fix, I would calmly explain the user impact and provide examples of how it might affect user experience or trust. If the impact is truly small, I would accept a lower priority but ensure it is properly documented and scheduled for later review instead of being ignored.

**How can testers ensure that low-priority bugs don’t get forgotten?**
To ensure low-priority bugs are not forgotten, testers can make sure they are logged clearly in the tracking system, tagged correctly, and linked to future releases or backlog items. During sprint planning or backlog refinement, QA can remind the team about recurring or grouped low-priority issues. Regular backlog review helps prevent minor issues from being permanently ignored.
