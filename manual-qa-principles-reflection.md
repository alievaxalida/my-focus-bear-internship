**What is manual QA testing, and how does it differ from automated testing?**
Manual QA testing is the process where a human tester interacts with the software to verify that it works correctly and meets user expectations. The tester explores the application, checks functionality, observes behavior, and evaluates usability. Unlike automated testing, which runs predefined scripts using tools, manual testing relies on human observation, critical thinking, and flexibility. Automation is fast and repeatable, while manual testing is better for discovering unexpected problems, usability issues, and edge cases.

**What is context-driven testing, and how does it challenge traditional QA approaches?**
Context-driven testing is an approach that says there is no single “best” way to test software. Instead, testing methods should adapt to the specific context of the project. Factors such as the product type, users, risks, deadlines, and team structure influence how testing should be done. This idea challenges traditional QA approaches that rely heavily on rigid processes, strict documentation, and fixed test plans. Context-driven testing encourages testers to think critically and adjust their approach based on the situation.

**What are the seven principles of context-driven testing?**
The seven principles of context-driven testing are commonly summarized as follows:
The value of any testing practice depends on its context.
There are good practices in testing, but no universal best practices.
People working together are the most important part of a project’s context.
Projects evolve over time, and testing must adapt to those changes.
Testing is an intellectual activity that requires skill and judgment.
Only through experimentation and experience can we know if testing was effective.
The goal of testing is to provide valuable information to help stakeholders make decisions.

**How do exploratory testing and scripted testing differ, and when should each be used?**
Exploratory testing and scripted testing are two different approaches. Exploratory testing is flexible and involves learning about the system while testing it. The tester actively explores features, asks questions, and investigates unusual behavior. Scripted testing follows predefined steps and expected results. It is useful for regression testing and ensuring consistent coverage. Exploratory testing is best for discovering new issues and understanding complex features, while scripted testing is better for repeating known checks.

**What skills make a great manual tester, according to James Bach and other experts?**
According to experts like James Bach, great manual testers need strong critical thinking skills, curiosity, attention to detail, good communication skills, and the ability to think like a user. They should also understand risk, ask good questions, and be comfortable exploring systems without strict instructions.

📝 Reflection

**Why is context important in testing, and how might testing Focus Bear differ from testing another app?**
Context is important in testing because different products have different risks, users, and expectations. Testing Focus Bear might focus more on usability, focus management, and smooth workflows, since it helps people manage attention and productivity. Testing another app, such as a financial system, would require stronger focus on security, accuracy, and compliance.

**How can manual testing be both structured and adaptable at the same time?**
Manual testing can be both structured and adaptable by combining test cases with exploratory sessions. Structured testing ensures that important scenarios are covered, while exploratory testing allows testers to investigate unexpected behavior and edge cases.

**How would you approach testing a new feature in Focus Bear without a predefined test script?**
If I were testing a new feature in Focus Bear without a predefined script, I would first try to understand the purpose of the feature and the main user flow. Then I would test the core functionality, check how it behaves under normal conditions, try incorrect inputs, explore edge cases, and observe the user experience. I would also check how the new feature interacts with existing parts of the application.

**How do you decide what to test when no detailed requirements are provided?**
When no detailed requirements are provided, I would decide what to test by focusing on risk and user impact. I would start with the core functionality, recent changes, and areas that could break easily. I would also compare the feature with similar parts of the app to check for consistency and expected behavior.
