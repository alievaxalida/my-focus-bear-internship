🏆 Milestone: Reporting & Tracking Issues Effectively
📝 How to Write Effective GitHub Issues
🎯 Goal
Learn how to create well-structured GitHub issues that provide clear, actionable details for the development team.

❓ Why is this important?
A well-written issue ensures that developers understand the problem quickly, reducing back-and-forth communication and improving resolution time. Poorly written issues can lead to confusion, delays, or even ignored bugs.

🔍 Research & Learn

**What are the key components of a good GitHub issue?**
The key components of a good GitHub issue are a clear and specific title, a short but complete description of the problem, steps to reproduce, expected result, actual result, environment details (device, OS, app version, browser), and attachments such as screenshots, logs, or screen recordings. The issue should also describe the impact on users. Clear formatting and simple language help developers understand the problem quickly.

**What is the difference between a bug report, feature request, and technical task?**
A bug report describes something that is not working as expected. A feature request suggests new functionality or an improvement. A technical task usually refers to internal work such as refactoring code, updating dependencies, or improving performance. It may not describe a visible problem but rather development work that needs to be done.

**How do labels, milestones, and assignees help organize issues in GitHub Projects?**
Labels, milestones, and assignees help organize work in GitHub Projects. Labels categorize issues (bug, enhancement, high priority, backend, frontend). Milestones group issues related to a specific release or sprint. Assignees show who is responsible for working on the issue. This structure helps teams track progress and avoid confusion.

**What are some examples of well-written vs. poorly written GitHub issues?**
A poorly written issue might say: “App crashed. Please fix.” It gives no details, no steps, and no environment information. A well-written issue clearly explains when the crash happens, how to reproduce it, what was expected, what actually happened, and includes logs or a recording.

📝 Reflection 

**How would you write a bug report for a crash in Focus Bear?**
If I were writing a bug report for a crash in Focus Bear, I would create a title like: “App crashes when starting Focus Session after login on Windows 10.” Then I would list the steps: log in, click ‘Start Focus Session,’ wait 3 seconds. I would write the expected result (Focus session starts normally) and the actual result (App closes unexpectedly). I would include OS version, app version, and attach logs or a screen recording.

**How can adding steps to reproduce help developers fix issues faster?**
Adding steps to reproduce helps developers fix issues faster because they can follow the same path and see the exact problem. Without clear steps, developers may test the wrong scenario and not see the issue. 

**If a developer asks for more details on an issue you created, what might you have missed?**
If a developer asks for more details, I might have missed environment information, exact steps, frequency of the issue, logs, or clarification about expected behavior. It usually means the report was not detailed enough for them to reproduce the problem confidently.
